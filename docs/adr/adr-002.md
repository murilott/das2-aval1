# ADR-002: Estratégia de Armazenamento de Dados

**Status:** Aceito | **Data:** 2026-04-03 | **Autores:** João Binelo, Thiago Lessenko

---

## Contexto

A plataforma de Inteligência Operacional da VendaMais precisa armazenar dois tipos distintos de dados ao longo do pipeline:

1. **Dados brutos (raw)**: arquivos extraídos diretamente do ERP, sem nenhuma transformação, preservados para reprocessamento e auditoria.  
2. **Dados tratados (curated/analíticos)**: registros normalizados, com regras de negócio aplicadas, prontos para consumo pelo Power BI.

Os requisitos que guiam esta decisão são:

- Rastreabilidade completa: deve ser possível reprocessar qualquer extração histórica a partir dos arquivos originais.  
- Histórico de dados: o SOW define a janela mínima de histórico a partir de janeiro de 2024.  
- Qualidade e confiabilidade: o Power BI é a camada de consumo final, os dados expostos devem ser consistentes, relacionados e sem duplicatas.  
- Defasagem máxima de 24 horas: a solução de armazenamento não pode ser gargalo de performance.  
- Custo: a VendaMais é uma empresa de médio porte; soluções de alto custo fixo precisam ser justificadas pelo valor entregue.  
- A plataforma é Microsoft Azure (restrição de contrato, SOW-2026-001).

O volume estimado é baixo-moderado: 4 módulos × extração diária × ~3.500 pedidos/mês, com dados de clientes, representantes, SKUs, títulos financeiros e logística.

---

## Decisão

Adotar uma **arquitetura de armazenamento em duas camadas complementares**:

**Camada 1 - Azure Blob Storage (dados brutos / raw)**  
Os arquivos extraídos do ERP pela Azure Function de ingestão são depositados no Blob Storage em formato JSON ou CSV, organizados por caminho de partição temporal: `/raw/{modulo}/YYYY/MM/DD/{arquivo}`. Nenhuma transformação é aplicada nesta camada, o objetivo é preservar os dados exatamente como vieram do ERP, garantindo rastreabilidade e possibilidade de reprocessamento.

Uma segunda subpasta (`/curated/`) armazena os arquivos intermediários após limpeza inicial, antes da carga no banco relacional.

**Camada 2 - Azure SQL Database (dados tratados / analíticos)**  
Após transformação pela segunda Azure Function, os dados normalizados são persistidos em tabelas relacionais no Azure SQL Database (tier Basic ou Standard S1, escalável conforme crescimento). O banco serve como **fonte única de verdade (Single Source of Truth)** para todos os dashboards do Power BI, garantindo consistência entre áreas.

O modelo de dados do Azure SQL reflete os quatro módulos: `fato\_vendas\`, `fato\_estoque\`, `fato\_financeiro\`, `fato\_logistica\`, com dimensões compartilhadas (`dim\_cliente\`, `dim\_representante\`, `dim\_produto\`, `dim\_data\`).

Credenciais de acesso ao banco são gerenciadas exclusivamente via Azure Key Vault e Managed Identity, nunca armazenadas no código ou no repositório.

---

## Consequências

(+) **Separação de responsabilidades clara**: O Blob Storage cuida da persistência durável e barata dos dados brutos; o Azure SQL cuida da estrutura relacional e da qualidade analítica. Cada camada faz o que faz melhor.

(+) **Rastreabilidade e reprocessamento**: Como os arquivos raw são preservados no Blob Storage, qualquer falha de transformação pode ser corrigida e reprocessada sem precisar re-extrair do ERP, evitando dependência de disponibilidade do sistema legado.

(+) **Power BI integrado nativamente ao Azure SQL**: O conector direto entre Power BI Service e Azure SQL Database elimina a necessidade de configurações extras de gateway para dados on-premises, simplificando a atualização agendada dos dashboards.

(+) **Custo previsível e controlado**: O Azure Blob Storage tem custo por GB armazenado, extremamente baixo para o volume da VendaMais. O Azure SQL no tier Basic cobre os requisitos iniciais com custo fixo mensal reduzido, escalável sob demanda.

(+) **Consistência e integridade referencial**: O modelo relacional no Azure SQL garante integridade entre módulos (ex.: um pedido de venda sempre referencia um cliente e um representante existentes), reduzindo a chance de dashboards com dados inconsistentes.

(-) **Dupla persistência**: Os dados transitam por duas camadas de armazenamento, o que aumenta marginalmente o custo total. Para o volume da VendaMais, o impacto é irrelevante, mas deve ser revisitado em cenários de crescimento acelerado.

(-) **Gerenciamento de schema do Azure SQL**: Alterações no ERP que modifiquem colunas ou entidades exigirão migrations no banco relacional e atualização das transformações. É necessário um processo de controle de mudança de schema (ex.: Flyway ou scripts versionados no Git).

---

## Alternativas rejeitadas

- **Azure SQL Database como único armazenamento (sem Blob Storage)**: Simplifica a arquitetura, mas elimina a camada raw. Sem os arquivos brutos, qualquer erro de transformação exige re-extração do ERP, gerando dependência do sistema legado e risco de perda de dados históricos. Rejeitado por comprometer a rastreabilidade exigida no OBJ-02 do SOW.

- **Azure Cosmos DB (NoSQL) no lugar do Azure SQL**: Banco de documentos NoSQL com escalabilidade global e schema flexível. Rejeitado porque o modelo de dados da VendaMais é intrinsecamente relacional (pedidos -> clientes -> representantes -> produtos), e Cosmos DB exige desnormalização e duplicação de dados para consultas analíticas eficientes, aumentando a complexidade de manutenção sem benefício para o caso de uso. O Power BI também tem integração mais madura e direta com SQL.

- **Azure Synapse Analytics (Data Warehouse)**: Plataforma completa de analytics com suporte a grandes volumes, processamento distribuído e integração nativa com Power BI. Rejeitado porque o volume de dados da VendaMais (4 módulos, extração diária, médio porte) está muito abaixo do threshold onde Synapse entrega valor diferencial. O custo fixo e a complexidade operacional de um Data Warehouse são desproporcionais ao problema atual. Pode ser considerado em uma futura evolução da plataforma se o volume crescer significativamente.

- **Azure Data Lake Storage Gen2 (ADLS) como camada analítica**: Solução de Data Lake com suporte a hierarquia de namespaces e integração com Spark/Databricks. Rejeitado pela mesma razão que o Synapse: o overhead de configuração e custo de processamento distribuído não se justifica para o volume atual. ADLS é adequado quando há necessidade de processamento massivo em paralelo ou múltiplas fontes heterogêneas.

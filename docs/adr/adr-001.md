\# ADR-001: Estratégia de Ingestão de Dados do ERP

\*\*Status:\*\* Aceito | \*\*Data:\*\* 2026-04-03 | \*\*Autores:\*\* João Binelo, Thiago Lessenko

\---

\#\# Contexto

A VendaMais Distribuidora Ltda. opera um ERP proprietário desde 2019 que centraliza transações de Vendas, Estoque, Financeiro e Logística. O sistema processa aproximadamente 3.500 pedidos/mês com 1.200 clientes ativos e 18 representantes comerciais.

O problema central é que a geração de relatórios é inteiramente manual, exportações pontuais em Excel que levam até 2 dias úteis para serem compiladas, resultando em dados com até 30 dias de defasagem para a diretoria. O projeto tem como requisito entregar dados com defasagem máxima de 24 horas em relação à origem, sem intervenção manual.

A camada de ingestão é a porta de entrada de toda a plataforma: ela deve conectar no ERP, extrair os registros do dia e depositá-los no Azure Blob Storage. As restrições relevantes para esta decisão são:

\- A solução deve rodar sobre a plataforma Microsoft Azure (já definido no SOW).  
\- O volume de dados é previsível e relativamente baixo (extração diária de 4 módulos: Vendas, Estoque, Financeiro, Logística).  
\- Não são permitidas modificações no ERP ou em sua base de dados.  
\- O orçamento é condizente com uma equipe de médio porte; soluções de alto custo fixo devem ser justificadas.  
\- A janela de execução é periódica (diária), não requerendo processamento contínuo de stream.

\---

\#\# Decisão

Adotar Azure Functions (Python) com gatilho de Timer (Time Trigger / Cron Schedule) como mecanismo de ingestão.

Cada módulo de dados (Vendas, Estoque, Financeiro, Logística) terá uma função dedicada, acionada diariamente em horário configurável (ex.: 02:00h). A função conecta via API REST ou query direta ao banco do ERP, extrai os registros do período, serializa em JSON/CSV e deposita os arquivos na camada \`raw\` do Azure Blob Storage, organizados por módulo e data (\`/raw/{modulo}/YYYY/MM/DD/\`).

O código das funções é versionado no repositório Git do projeto. As credenciais de acesso ao ERP são armazenadas exclusivamente no \*\*Azure Key Vault\*\*, referenciadas nas funções por variável de ambiente, nunca hardcoded no código.

\---

\#\# Consequências

(+) \*\*Custo zero em idle\*\*: Azure Functions no plano Consumption cobra apenas pelo tempo de execução. Como as funções rodam uma vez por dia e por poucos minutos, o custo mensal é próximo de zero (dentro do free tier de 1 milhão de execuções/mês).

(+) \*\*Sem infraestrutura a gerenciar\*\*: Não há servidor, VM ou container para provisionar, monitorar ou patchear. A plataforma Azure gerencia escalonamento, disponibilidade e runtime.

(+) \*\*Deploy e manutenção simples\*\*: Funções Python são simples de escrever, testar localmente com \`func start\` (Azure Functions Core Tools) e fazer deploy via GitHub Actions. O código fica versionado junto ao restante do projeto.

(+) \*\*Rastreabilidade nativa\*\*: Integração automática com Azure Application Insights para logs, rastreamento de falhas e alertas, essencial para auditar se a ingestão do dia ocorreu com sucesso.

(-) \*\*Cold start\*\*: A primeira execução após período inativo pode ter latência de alguns segundos. Para gatilhos de timer diário isso é irrelevante, mas é um ponto a observar caso futuras expansões exijam resposta em tempo real.

(-) \*\*Timeout máximo de 10 minutos (plano Consumption)\*\*: Se o volume de dados crescer significativamente e a extração ultrapassar esse limite, será necessário migrar para o plano Premium ou refatorar a função em partes menores. Hoje o volume não representa risco.

(-) \*\*Dependência de conectividade com o ERP\*\*: Se o ERP estiver indisponível no horário agendado, a extração falha. É necessário implementar política de retry e monitoramento de alertas para reprocessamento manual em caso de falha.

\---

\#\# Alternativas rejeitadas

\- \*\*Azure Data Factory (ADF)\*\*: Serviço gerenciado de ETL/ELT com conectores nativos para diversas fontes. Rejeitado porque o custo fixo mensal e a complexidade de configuração (pipelines, linked services, Integration Runtime) são desproporcionais ao volume e à simplicidade do caso de uso da VendaMais. ADF é mais adequado para migrações em larga escala ou integrações com dezenas de fontes heterogêneas.

\- \*\*Azure Logic Apps\*\*: Plataforma low-code de automação de fluxos com gatilhos de timer. Rejeitado porque, apesar de simples de configurar, oferece menos controle sobre lógica de extração customizada, tratamento de erros e versionamento de código. Para conexões com ERP proprietário que podem exigir autenticação específica e parsing complexo, Python em Azure Functions é superior.

\- \*\*Script Python agendado em Azure VM (Cron)\*\*: Solução tradicional de script rodando em uma máquina virtual. Rejeitado porque exige provisionamento, gerenciamento e custo contínuo de uma VM, mesmo que ela fique ociosa 23h55min por dia. Vai contra o princípio de eficiência de custos e aumenta a superfície de operação da equipe.
